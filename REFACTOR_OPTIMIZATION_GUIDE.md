# SmartFin 结构优化与重构指南（不改业务功能）

> 目标：在**不改变现有业务逻辑和用户行为**的前提下，逐步优化代码结构、模块边界和实现质量，降低维护成本与回归风险。  
> 适用时机：完成阶段性功能开发后，按本指南分批执行。

---

## 1. 当前项目状态评估（工程视角）

### 1.1 已有优势

- 已具备模块化骨架：`src/lib/server/modules/*`（`api/service/repository/events`）。
- OCR 能力较完整：包含文本提取、LLM结构化、字段归一化、启发式规则。
- 已有运行时模块开关能力（`enabled.ts`）与模块注册机制。

### 1.2 主要结构问题（需优先治理）

- **路由层旁路模块层**：大量 `+page.server.ts` / `+server.ts` 仍直接依赖 `legacy-db`。
- **边界规则与实际代码不一致**：与 `MODULARIZATION_GUIDE.md` 的目标存在偏差。
- **职责混杂**：例如 `expense` API 同时承载 revenue 能力，模块语义不清。
- **平台策略分散**：路径鉴权、模块映射、模块启用判断散落多处。
- **错误处理不统一**：不同 API 返回结构与错误语义不一致。
- **超长 Handler**：如文档上传保存入口承担过多分支与职责，维护成本高。
- **类型边界偏弱**：事件总线存在强制断言，事件 payload 契约不够强类型。
- **自动化测试薄弱**：缺乏重构保护网。

---

## 2. 重构总原则（必须遵守）

- **原则A：零功能变更**
  - 不改业务规则、不改接口语义、不改数据库行为（除非明确进入功能迭代）。
- **原则B：先收敛边界，再优化实现**
  - 先解决“代码放错层”的问题，再做内部重构。
- **原则C：小步提交、可回滚**
  - 单次改造控制在一个主题（例如“错误处理统一”）内。
- **原则D：建立护栏防止回流**
  - 新代码禁止继续引入 `legacy-db`。
- **原则E：每一步都有验收标准**
  - 结构改造必须可验证（lint、类型检查、关键路径回归）。

## 3. 分阶段优化路线图

## 阶段1（1-3天）：低风险高收益收敛

### 3.1 统一 API 错误处理协议

- 目标：
  - 统一错误输出形态，例如：`{ ok: false, code, message, details? }`。
- 操作：
  - 新增统一错误包装器（如 `withApiErrorHandling`）。
  - 将热点 API 逐步接入（先从 OCR、费用上传、文档上传开始）。
- 验收：
  - 同类错误返回结构一致。
  - 前端错误提示无需为不同 API 写特判。

### 3.2 拆分超长 Handler（只拆结构，不改逻辑）

- 目标：
  - 降低单文件复杂度，减少分支耦合。
- 操作：
  - 将 `save-project-document` 按 `docType` 拆分为独立处理器函数/文件。
  - 保持字段映射、默认值、写库逻辑完全一致。
- 验收：
  - 输入输出一致（可通过回放请求样本验证）。
  - 单文件复杂度明显下降。

### 3.3 防止债务继续增加

- 目标：
  - 禁止新增路由直接依赖 `legacy-db`。
- 操作：
  - 增加 lint/review 规则（PR 检查项）。
  - 在团队约定中明确“新路由必须走 module API/query service”。
- 验收：
  - 新提交不再新增 `legacy-db` 依赖点。

---

## 阶段2（1-2周）：边界清晰化与类型安全

### 4.1 收敛路由职责

- 目标：
  - 路由层仅负责解析请求、权限校验、调用模块 API。
- 操作：
  - 按优先级迁移高频路径：
    - `expenses/upload`
    - `documents/upload`
    - `dashboard/overview`
  - 迁移后删除对应路由中的 SQL/仓储细节。
- 验收：
  - 路由层不再出现直接 DB 读写（除极少临时兼容点）。

### 4.2 统一平台策略配置（鉴权 + 模块映射）

- 目标：
  - 消除多处路径前缀硬编码。
- 操作：
  - 抽出单一“路由能力声明表”，统一驱动：
    - 是否需要登录
    - 所属模块
    - 是否受模块开关控制
- 验收：
  - 新增路由仅需改一处配置。
  - 不再出现“鉴权通过但模块被禁用/漏映射”的错配。

### 4.3 强化事件总线类型

- 目标：
  - 让事件名与 payload 在编译期受约束。
- 操作：
  - 建立 `EventMap`（事件名 -> payload 类型）。
  - `emit/on` 改为泛型绑定，逐步移除 `unknown/as` 强转。
- 验收：
  - 事件字段变更可触发编译报错，而非运行时错误。

### 4.4 模块职责拆分

- 目标：
  - `expense` 与 `revenue` 职责清晰，减少语义混淆。
- 操作：
  - 先做 API 层拆分（兼容旧入口），再逐步拆 service/repository。
- 验收：
  - 代码命名与业务域一致。
  - 新代码不再把 revenue 逻辑放入 expense 模块。

---

## 阶段3（2周以上）：工程治理与可持续演进

### 5.1 实现“路由零直连 DB”

- 目标：
  - 所有读写通过模块 API / query service 完成。
- 验收：
  - 可移除 `legacy-db`（或仅保留极短暂兼容壳）。

### 5.2 建立最小测试矩阵（重构护栏）

- 建议最小覆盖：
  - 模块 API 单测（核心写操作）。
  - 关键 API 契约测试（错误结构、状态码、响应字段）。
  - OCR 处理链回归样本测试（避免字段回归）。
- 验收：
  - 重构类 PR 必须通过测试矩阵后合并。

### 5.3 清理废弃入口与兼容桥

- 目标：
  - 去除 `deprecated` 包装和失效端点噪音。
- 验收：
  - 对外入口清晰，维护者不再误用旧路径。

---

## 4. 优先级与执行顺序（建议）

1. 统一错误协议（快、收益高、几乎无业务风险）  
2. 拆超长 Handler（减复杂度）  
3. 迁移高频路由到模块层（控制最大风险源）  
4. 收敛路径策略配置（减少后续漏改）  
5. 事件总线类型化（提升长期可靠性）  
6. 模块职责拆分（expense vs revenue）  
7. 建测试护栏 + 清理兼容债务

---

## 5. 每次重构提交的验收清单（Checklist）

- [ ] 无功能改动（接口行为、状态码、字段语义一致）  
- [ ] `pnpm lint` / `npm run lint` 通过  
- [ ] `pnpm check` / `npm run check` 通过  
- [ ] 关键页面与关键 API 手工回归通过  
- [ ] 无新增 `legacy-db` 依赖  
- [ ] 错误返回结构符合统一约定  
- [ ] PR 描述包含“改造前后边界变化说明”

---

## 6. 风险控制建议

- 每次只做一种类型改造（例如“只做错误处理统一”）。
- 对高风险迁移点保留短期适配层（最多1-2迭代），避免一次性硬切。
- 在日志中增加 `correlationId` 贯穿请求链，便于重构期间排障。
- 先迁移“最常改、最易出错”的路径，再清理长尾。

---

## 7. 你可以直接采用的重构目标定义（DoD）

当满足以下条件时，可认为“结构优化第一阶段成功”：

- 路由层新代码不再直接读写 DB；
- 至少 3 个关键 API 已完成统一错误协议；
- 至少 2 个超长 Handler 完成拆分且行为一致；
- 路由鉴权与模块映射配置实现单源化；
- 重构 PR 具备最小测试与回归说明。
